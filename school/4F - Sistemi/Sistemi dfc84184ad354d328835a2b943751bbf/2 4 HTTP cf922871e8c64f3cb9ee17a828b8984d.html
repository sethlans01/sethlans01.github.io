2.4 HTTP

HTTP (HyperText Transfer Protocol) regola lo scambio di messaggi tra client e server web.

Versioni di HTTP

La prima versione HTTP/1.0 aveva un po’ di limitazioni:

  • mancanza di meccanismi di sicurezza, poiché non erano previste autenticazione e crittografia dei dati
  • non era possibile ospitare più siti web sullo stesso server
  • per ogni richiesta era necessario creare una connessione separata con il server

HTTP/1.1 permette di usare la stessa connessione per fare operazioni multiple.

HTTP/2 utilizza ulteriori misure di sicurezza a livello di trasporto, utilizzando TLS (Transport Layer Security).

HTTP/3 utilizza il protocollo UDP invece del protocollo TCP.

Come funziona HTTP

Ogni pagina web ha un suo indirizzo simbolico, detto URL (Uniform Resource Locator), come https://www.leagueoflegends.com/it-it/news/game-updates/patch-13-21-notes/.

Per tradurre un URL nel corrispondente indirizzo IP, si utilizza il DNS (Domain Name System).

Un URL è diviso in varie parti:

http://www.leagueoflegends.com/it-it/news/game-updates/patch-13-21-notes/
Indica al browser quale protocollo deve essere usatoidentifica il nome di una specifica macchina (il web server)rappresenta l’entità di dominio del sito webidentifica il path del file che deve essere mostrato. Se non viene specificato alcun file, il browser carica la pagina web di default presente sul server.

HTTP è un protocollo stateless, cioè non tiene traccia della connessione appena instaurata, rendendo così possibile saltare velocemente da un web server ad un altro, attraverso i link ipertestuali.

Ad ogni richiesta di un web client, HTTP effettua una nuova connessione al server, che viene chiusa al termine del trasferimento dell’oggetto richiesto.

Il protocollo HTTP lavora sulla porta 80.

Poiché HTTP è un protocollo stateless, ci serve un modo per memorizzare i dati di un utente che effettua più richieste di fila allo stesso server: ad esempio, è importante trovare un modo per salvare i dati di login di. un utente.

Per questo motivo si utilizzano i cookie, che sono piccoli blocchi di dati memorizzati nel browser e che permettono di:

  • implementare metodi di autenticazione
  • memorizzare dati utili di una sessione di navigazione
  • tracciare la navigazione dell’utente, per fini statistici o pubblicitari

Metodi e messaggi di HTTP

L’acquisizione di una risorsa da parte di un client può essere schematizzata in 4 fasi:

  1. Connessione: il client crea una connessione TCP/IP con il server, usando il suo nome di dominio o il suo indirizzo IP e il numero di porta.
    1. Se il numero di porta non viene specificato, viene usata la porta di default 80
  1. Richiesta: il client richiede la risorsa desiderata
  1. Risposta: il server invia una risposta al client, che può contenere i dati richiesti oppure un messaggio di errore
  1. Disconnessione: il server, dopo aver inviato la sua risposta, si disconnette dal client.

HTTP mette a disposizione del client dei metodi, che sono comandi che il client manda al server.

I metodi a disposizione sono i seguenti:

  • GET: il client richiede una risorsa al server
  • POST: il client invia informazioni o dati al server
  • HEAD: il client richiede al server solo l’header, senza la risorsa. Si usa per la diagnostica.
  • PUT: richiede l’upload di un file al server
  • DELETE: richiede la cancellazione di un file dal server
  • OPTIONS: richiede l’elenco dei metodi permessi dal server
  • TRACE: traccia una richiesta, visualizzando come viene gestita dal server
  • CONNECT: richiede una connessione mediante proxy

Richieste HTTP

Una richiesta HTTP è composta dalle seguenti 3 parti:

  1. riga di richiesta (request line)
  1. sezione header (contiene informazioni aggiuntive)
  1. body (contenuto della richiesta)

La request line è composta da metodo, URI e versione del protocollo.

Un URI (Uniform Resource Identifier) è un identificativo unico utilizzato per indicare una qualsiasi risorsa.

Un esempio di request line è il seguente:

GET /info.html HTTP/1.1

Gli header più comuni sono:

  • Host: nome del server a cui si riferisce l’URI
  • User-Agent: identificazione del tipo di client: browser, produttore, versione….

Risposte HTTP

Una risposta HTTP è composta dalle seguenti 3 parti:

  1. Riga di stato: contiene un codice di risposta a 3 cifre in cui la prima cifra indica il tipo di stato
    • 1xx: Informational, messaggi informativi
    • 2xx: Success, la richiesta è stata soddisfatta
    • 3xx: Redirection, la richiesta è considerata corretta, ma non c’è una risposta diretta. Viene indicato come ottenere la risposta
    • 4xx: Client Error, la richiesta non può essere soddisfatta perché è sbagliata
    • 5xx: Server Error, la richiesta non può essere soddisfatta per un problema interno del server
  1. header: contiene informazioni aggiuntive.
    • Server: indica il tipo e la versione del server
    • Content-Type: specifica il tipo di contenuto inviato il client
  1. body: parte in cui si trova il contenuto della risposta.

I codici di stato (status code) più comuni sono:

  • 200 OK: il server ha fornito il contenuto richiesto, che si trova nella sezione body
  • 400 Bad Request: il server non capisce la richiesta
  • 403 Forbidden: non si possiede il livello adeguato di permessi per poter accedere alla risorsa richiesta
  • 404 Not Found: il client ha richiesto una risorsa che non esiste sul server
  • 500 Internal Server Error: il server non riesce a rispondere alla richiesta, a causa di un problema interno
  • 505 HTTP Version Not Supported: la versione di HTTP usata nella richiesta non è supportata dal server

Proxy HTTP

Un proxy è un programma che si interpone tra client e server, facendo da tramite.

Il client si collega al proxy, invece che al server, e gli invia le richieste.

Il proxy, a sua volta, si collega al server, gli inoltra le richieste ricevute dai client e inoltra ai client le risposte ricevute dal server.

Un proxy si usa per vari motivi:

  • Connettività: un server proxy può essere usato per permettere ad una rete privata di accedere direttamente ad Internet con un unico computer, invece di connettersi con tutti i computer in maniera diretta.
  • Privacy: è possibile mascherare il vero indirizzo IP del client, in modo che il server remoto non conosca l’indirizzo di chi ha effettuato la richiesta.
  • Caching: un server proxy può immagazzinare per un po’ di tempo i risultati delle richieste dei client e se un altro client effettua le stesse richieste, può rispondere senza contattare il server originale.
  • Monitoraggio e amministrazione: è possibile monitorare e filtrare le richieste effettuate dai vari client connessi al proxy.
