1.2 Le Repository

Creare una repository

In genere si crea un repository Git in due modi:

  1. Si può prendere una directory locale che non è attualmente sotto VCS e trasformarla in un repository Git.
  1. Si può clonare un repository Git esistente da un altro luogo.

In entrambi i casi, si ottiene un repository Git sulla macchina locale, pronta per il lavoro.

Inizializzare un repository in una directory esistente

Se si dispone di una directory di progetto che attualmente non è sotto controllo di versione e si vuole iniziare a controllarla con Git, occorre innanzitutto andare nella directory del progetto.

Se non l'avete mai fatto, l'aspetto è un po' diverso a seconda del sistema in uso:

cd /c/user/my_project //Dipende dalla directory in cui volete creare il repository

O potete farlo via interfaccia grafica.

Questo crea una nuova sottodirectory denominata .git che contiene tutti i file della repository necessari - uno scheletro del repository Git.

A questo punto, nulla del progetto è ancora tracciato.

Se si vuole iniziare a controllare la versione di file esistenti (invece di una directory vuota), si dovrebbe iniziare a tracciare quei file e fare un commit iniziale.

È possibile farlo con i seguenti comandi:

git add *.html
// Questo comando aggiunge nella staging area tutti i file .html della cartella

git commit -m 'versione iniziale del progetto'
// Questo comando esegue il commit, con il messaggio specificato tra i due apici

Clonare una repository esistente

Se si vuole ottenere una copia di un repository Git esistente, per esempio un progetto a cui si vuole contribuire, il comando necessario è git clone.

Si clona una repository con git clone <url>. Per esempio, se si vuole clonare la libreria linkabile Git chiamata libgit2, si può fare così:

git clone https://github.com/libgit2/libgit2

Questo crea una directory chiamata libgit2, inizializza una directory .git all'interno di essa, estrae tutti i dati per quella repository e crea una copia funzionante dell'ultima versione.

Se si entra nella nuova cartella libgit2 appena creata, si vedranno i file del progetto, pronti per essere lavorati o utilizzati.

Se si vuole clonare il repository in una directory con un nome diverso da libgit2, si può specificare il nome della nuova directory come argomento aggiuntivo:

git clone https://github.com/libgit2/libgit2 mylibgit

Modifiche alle repository

Ricordate che ogni file nella vostra directory di lavoro può trovarsi in uno dei due stati: tracciato o non tracciato.

I file tracciati sono quelli presenti nell'ultimo snapshot; possono essere non modificati, modificati o staged.

In breve, i file tracciati sono i file di cui Git è a conoscenza.

I file non tracciati sono tutto il resto: qualsiasi file nella vostra directory che non era nell'ultimo snapshot e non è nella staging area.

Quando si clona per la prima volta un repository, tutti i file saranno non modificati, perché Git li ha appena controllati e voi non avete modificato nulla.

Quando si modificano i file, Git li vede come modificati, perché sono stati modificati dall'ultimo commit.

Mentre si lavora, si mettono nella staging area questi file modificati e poi si esegue il commit di tutte le modifiche messe in staging area, e il ciclo si ripete.

Tracciare nuovi file

Per iniziare a tenere traccia di un file si usa il comando git add.

git add index.html
// Con questo comando, lo stato di index.html passa da untracked a tracked.

Questo comando vuole come argomento un path o verso un file o verso una cartella: se si passa come argomento il path di una cartella, verranno aggiunti automaticamente nella staging area tutti i file di quella cartella.

Se usate il comando git status di nuovo, potete vedere che il file index.html è ora tracciato e nella staging area.

$ git status


On branch master
Your branch is up-to-date with 'origin/master'.
Changes to be committed:
   (use "git reset HEAD <file>..." to unstage)
     new file: index.html

Si può capire che index.html è staged, dato che si trova nella sezione ”Changes to be committed”.

Se eseguite un commit, la versione del file che verrà messa nello snapshot è quella che è stata aggiunta quando avete usato il comando git add.

Fare un commit

Ora che la staging area è impostata nel modo desiderato, è possibile eseguire il commit delle modifiche.

Ricordate che tutti i file unstaged, cioè tutti i file creati o modificati su cui non è stato eseguito git add da quando sono stati modificati, non verranno inseriti in questo commit.

Il modo più semplice per fare il commit è digitare:

git commit

Saltare la Staging Area

Se volete saltare la staging area, basta aggiungere il flag -a al comando di commit: così facendo Git aggiungerà in automatico alla staging area tutti i file che, prima del commit, erano tracked.

$ git commit -a -m 'added new benchmarks'

Il comando -m ‘nuovo commento’ permette di inserire un commento senza aprire un editor.

Rimuovere Files

Per rimuovere un file da Git, bisogna rimuoverlo dalla staging area e poi fare un commit.

Il comando git rm rimuove il file dalla staging area e dalla directory.

Attenzione: dopo aver fatto git rm è necessario eseguire un commit per rendere effettiva l’eliminazione!!

git rm index.html
git commit -m "Rimosso index.html"

Se avete modificato un file e lo avete già aggiunto alla staging area, ma (per qualche motivo) volete rimuoverlo da lì, basta aggiungere il flag -f al comando di rimozione.

/* IERI */
// Modifico il file index.html
git add index.html               // Lo aggiungo alla staging area

/* OGGI */
// Voglio rimuovere il file dalla staging area
git rm -f index.html

Vedere la Commit History

Se volete vedere la storia dei commit eseguiti fino ad ora, usate il seguente comando:

git log
